03 · Task Analysis

회의 시간 정하기, 정말 ‘하나의 일’일까요?

회의 시간 정하기

①

모으기

다들 언제 되는지 묻고, 응답을 하나씩 모아야 합니다.

②

비교하기

모인 시간을 비교해 가장 괜찮은 시간을 찾아야 합니다.

③

조율하기

모두가 가능한 시간이 없다면, 각자의 조건을 비교하고 수용 가능한 대안을 찾아야 합니다.

회의 시간 정하기는 하나의 일이 아니었습니다.

각자의 조건을 모으고, 가능한 시간을 비교하고, 의견을 조율하는
세 가지 일이 모두 주최자 한 사람에게 집중되어 있었습니다.

현재 서비스들은 이 부담을 어떻게 해결하고 있을까요?

What We Learned

기존 도구들은 모으기·비교하기·조율하기 중 일부만 지원했습니다.
세 과정을 하나의 흐름으로 연결해 주최자의 부담을 줄여주는 방식은 찾기 어려웠습니다.

그래서 세 과정을 하나의 흐름으로 연결하되,
시스템은 판단에 필요한 정보를 정리하고 주최자는 최종 결정을 내리는 구조로 설계했습니다.

05 · Design Direction

주최자가 맡던 세 가지 일을 시스템과 나눴습니다

기존 도구는 일부 과정을 도왔지만, 모으고·비교하고·조율하는 일은 여전히 주최자의 몫이었습니다.
필참 여부·선호·일정 제약까지 함께 고려해 시스템이 추천하고, 사람은 최종 결정에만 집중할 수 있도록 설계했습니다.

주최자는 시스템의 추천을 바탕으로 상황맥락을 고려해 최종 결정만 합니다.

06 · System Flow

나눈 역할을 회의 생성부터 확정까지 연결했습니다

  1. 주최자 회의 만들기
  2. 42시스템 회의 생성 메시지 전송
  3. 👥참석자 n 응답 제출
  4. 42시스템 응답 모으기 · 일정 비교하기 · 추천 시간 찾기
  5. 주최자 추천 결과 확인 · 최종 시간 결정
  6. 42시스템 확정 메시지 자동 전송
  7. 👥참석자 n 회의 확정 확인

회의 만들기부터 결과 확인까지, 각 주체의 행동이 하나의 흐름 안에서 이어집니다.

역할을 다시 나누는 데서 그치지 않고,
모으기·비교하기·조율하기 각 단계에서 사용자가 직접 해야 하는 일까지 줄였습니다.

07 · Build

설계한 구조를 직접 사용할 수 있는 프로토타입으로 구현했습니다

정보 구조를 설계한 뒤, Claude Code를 활용해 직접 조작할 수 있는 프로토타입으로 구현했습니다. 정지된 화면이 아니라 전체 흐름을 직접 따라가며, 화면이 아닌 실제 사용 경험을 확인했습니다.

처음부터 모든 화면을 완성하지 않았습니다. 핵심 흐름을 먼저 구현하고, 직접 사용하며 발견한 문제를 다음 버전에 바로 반영했습니다. AI로 구현과 수정 시간을 줄인 만큼, 사용 흐름을 검증하고 다음 방향을 판단하는 데 더 집중했습니다.

설계 → 프로토타입 → 검증 → 개선 ↺

빠르게 프로토타입을 만들고 검증하며 구체화 했습니다.

핵심 흐름부터 구현하고, 사용하며 발견한 문제를 다음 버전에 반영했습니다.

08 · Final Design

최종 경험 — 세 부담을 덜고, 지문의 단서를 닫았습니다

모으기·비교하기·조율하기를 시스템으로 다시 나눴습니다. 각 단계에서 입력·비교·판단 부담을 낮추는 동시에, 과제가 던진 단서들을 하나씩 해결했습니다.

주최자와 참석자, 양쪽 입력을 함께 줄입니다.

주최자 — 회의 이름과 후보 기간만 정하면 됩니다. 소요시간·응답 마감·참석자(전원 필참)는 기본값으로 오고, 요청은 채널에 자동 게시돼 따로 공지를 쓰지 않습니다.

참석자 — 캘린더를 자동으로 불러와 안 되는 시간만 표시합니다. ‘가능/불가’ 이분법과 달리 X(불가)와 ▲(피하고 싶어요)를 구분해 버려지던 선호까지 담습니다.

캘린더는 시작점, 정답이 아님 — 캘린더에 일정이 없는 사람(외근이라 안 적는 한소희)은 빈 그리드에서 직접 표시합니다. 한 사람이 캘린더에 없어도 조율이 멈추지 않습니다.

요청 자동 게시캘린더 자동 + 없으면 직접 보완외근 → X점심 직후 → ▲

후보를 다 늘어놓지 않고 최적의 하나만 크게 추천해 인지부담을 줄입니다. 더 보는 것도, 정하는 것도 사용자의 자유입니다.

응답이 모이면 시스템이 겹쳐 보고 “다들 참석할 수 있는 시간” 하나를 크게 제시합니다.

완벽한 시간은 없다 · 그래서 투명하게 — 모두가 편한 완벽한 시간은 없습니다. 그래서 추천은 늘 누가 ▲ 피하고 싶어했는지(트레이드오프)를 감추지 않고 함께 보여주고, 회피가 가장 적은 명확한 1등을 고릅니다. 추천이 자의적이지 않다는 근거입니다.

“다른 시간도 있어요(N)”로 언제든 전체를 펼쳐 볼 수 있고, “이 시간으로 정하기”는 사용자가 누릅니다.

6명 → 한눈에완벽한 시간 없음 → 트레이드오프 투명근거 있는 1등더보기·결정 = 사용자

조율은 시간을 정하는 일이자 감정을 쓰는 일이었습니다. 42는 판단의 근거와 함께 그 에너지까지 덜어줍니다. 완벽한 시간이 없어도 멈추지 않고, 최종 결정은 사람이 합니다.

성립 — 필참자가 모두 되는 시간을 확정하면 “다들 올 수 있는 괜찮은 시간으로 정했어요.”

성립 0 — 다 되는 시간이 없으면 억지로 만들지 않습니다. “다 되는 시간 없음”을 정직하게 알리고, 가까운 순 후보와 히트맵으로 사람이 직접 고르게 합니다.

양방향 조율 — 확정 후 누가 어려워지면 개별로 다시 묻지 않아도 됩니다. 참석자는 다른 시간 제안 / 나 없이 진행 / 개인 메시지 중에 고르고, 시스템은 받아둔 응답으로 ‘가장 적게 흔드는 순서’의 조정안을 중립적 문구로 올린 뒤 채널에 자동 공지합니다.

필수/선택 → 필참 전원 최선괜찮은 → “괜찮은 시간”성립 0 → 정직한 대안양방향 조율재메시지·완곡한 부탁 에너지 ↓

추천은 제안이고, 결정은 사람이 합니다.
