6명에게는 피하고 싶은 시간과 외근 일정이 있었고, 참석 우선순위도 달랐습니다.

What We Learned

도구마다 덜어준 일은 달랐어요. 하지만 끝까지 사람이 해야 하는 일은 남아 있었어요.

03 · Design Direction

주최자가 하던 일을 시스템이 맡도록 설계했습니다.

기존 도구는 일부 과정을 도왔지만, 모으고·비교하고·조율하는 일은 여전히 주최자의 몫이었습니다. 필참 여부·선호·일정 제약까지 함께 고려해 시스템이 추천하고, 사람은 최종 결정에만 집중할 수 있도록 설계했습니다.

시스템은 모으고, 비교하고, 추천합니다.

사람은 상황을 고려해 최종 결정합니다.

이 원칙을 바탕으로 회의를 만드는 경험을 다시 설계했습니다.

04 · System Flow

조율의 흐름을 시스템 중심으로 다시 설계했습니다.

회의 생성부터 확정까지, 반복되는 전달과 조율은 시스템이 담당하도록 정보 구조를 재구성했습니다. 주최자는 회의를 만들고 최종 결정만 내리며, 시스템은 참석 요청을 전달하고 응답을 모아 비교한 뒤 추천 시간을 제안하고, 확정 결과까지 자동으로 공유합니다.

  1. 🧑주최자 회의 만들기
  2. 🤖42 회의 생성 메시지 전송
  3. 👥참석자 1~6 응답 제출
  4. 🤖42 응답 모으기 · 일정 비교하기 · 추천 시간 찾기
  5. 🧑주최자 추천 결과 확인 · 최종 시간 결정
  6. 🤖42 확정 메시지 자동 전송
  7. 👥참석자 1~6 회의 확정 확인

🤖 42가 전달·수집·비교·공유를 모두 맡고, 사람(주최자)은 만들기·결정 두 지점만 개입합니다.

이제 각 단계를 실제 화면으로 살펴봅니다.

05 · Build · Test · Refine

하나의 시안이 아니라, 반복의 결과입니다.

정보 구조의 뼈대를 설계한 뒤, Claude Code를 활용해 실제 제품 형태로 구현하며 사용 경험을 검증했습니다. 구현 과정에서 발견한 문제를 수정하고, 사용자 테스트를 통해 받은 피드백을 반영하며 제품을 지속적으로 발전시켰습니다. 최종 결과물은 하나의 시안을 선택한 결과가 아니라, 설계 → 구현 → 검증 → 개선을 반복하며 사용자 경험을 완성한 결과입니다.

①

Build — Version History

동작하는 프로토타입은 목업이 아니라 실제 추천 로직 위에서 돌아갑니다. 메신저 안에서 그대로 쓰는 네이티브 경험으로 구현했습니다.

실제 구현 과정에서 제품이 발전한 주요 버전입니다.

②

Test — User Test

구현한 프로토타입으로 사용자 테스트를 진행하고, 주요 피드백을 반영해 최종 경험을 개선했습니다.
