03 · Task Analysis

회의 시간 정하기, 정말 ‘하나의 일’일까요?

회의 시간 정하기

①

모으기

다들 언제 되는지 묻고, 응답을 하나씩 모아야 합니다.

②

비교하기

모인 시간을 비교해 가장 괜찮은 시간을 찾아야 합니다.

③

조율하기

모두가 가능한 시간이 없다면, 각자의 조건을 비교하고 수용 가능한 대안을 찾아야 합니다.

회의 시간 정하기는 하나의 일이 아니었습니다.

각자의 조건을 모으고, 가능한 시간을 비교하고, 의견을 조율하는
세 가지 일이 모두 주최자 한 사람에게 집중되어 있었습니다.

현재 서비스들은 이 부담을 어떻게 해결하고 있을까요?

각 방식은 일부 과정만 도왔고, 모으기부터 조율하기까지 하나의 흐름으로 이어주지는 못했어요.

What We Learned

기존 도구들은 모으기·비교하기·조율하기 중 일부만 지원했습니다.
세 과정을 하나의 흐름으로 연결해 주최자의 부담을 줄여주는 방식은 찾기 어려웠습니다.

그래서 세 과정을 하나의 흐름으로 연결하되,
시스템은 판단에 필요한 정보를 정리하고 주최자는 최종 결정을 내리는 구조로 설계했습니다.

05 · Design Direction

주최자가 맡던 세 가지 일을 시스템과 나눴습니다

기존 도구는 일부 과정을 도왔지만, 모으고·비교하고·조율하는 일은 여전히 주최자의 몫이었습니다.
필참 여부·선호·일정 제약까지 함께 고려해 시스템이 추천하고, 사람은 최종 결정에만 집중할 수 있도록 설계했습니다.

회의 만들기부터 결과 확인까지, 각 주체의 행동이 하나의 흐름 안에서 이어집니다.

역할을 다시 나누는 데서 그치지 않고,
모으기·비교하기·조율하기 각 단계에서 사용자가 직접 해야 하는 일까지 줄였습니다.

07 · Build

설계한 흐름을 동작하는 프로토타입으로 발전시켰습니다

Claude Code로 회의 생성부터 확정까지 직접 사용할 수 있는 프로토타입을 구현했습니다. 정지된 시안에서는 알기 어려운 화면 간 연결과 상태 변화를 실제 흐름 안에서 빠르게 확인했습니다.

처음부터 모든 화면을 완성하지 않고, 핵심 흐름을 먼저 작동시킨 뒤, 직접 사용하며 발견한 문제를 다음 버전에 바로 반영했습니다.

08 · Final Design

최종 경험 — 세 부담을 덜고, 지문의 단서를 닫았습니다

모으기·비교하기·조율하기를 시스템으로 다시 나눴습니다. 각 단계에서 입력·비교·판단 부담을 낮추는 동시에, 과제가 던진 단서들을 하나씩 해결했습니다.

Final Design 데모 — 챕터별 화면

①모으기 · 입력 부담 ↓

주최자와 참석자, 양쪽 입력을 함께 줄입니다.

🎬회의 만들기(이름·기간, 나머지 기본값) → “만들고 #커머스팀에 요청 보내기” → 참석자 응답(캘린더 자동 / 미연동자는 빈 그리드, X·▲ 원탭)

요청 자동 게시캘린더 자동 + 없으면 직접 보완외근 → X점심 직후 → ▲

②비교하기 · 비교 부담 ↓ (인지부담까지)

후보를 다 늘어놓지 않고 최적의 하나만 크게 추천해 인지부담을 줄입니다. 더 보는 것도, 정하는 것도 사용자의 자유입니다.

🎬1/6→6/6 자동 채워짐 → 추천 카드 자동 등장(“나에게만 보여요”, ▲ 누가 피하고 싶어했는지) → “다른 시간도 있어요” 탭 → 전체 펼침(추천 배지 유지)

6명 → 한눈에완벽한 시간 없음 → 트레이드오프 투명근거 있는 1등더보기·결정 = 사용자

③조율하기 · 판단 부담 ↓

조율은 시간을 정하는 일이자 감정을 쓰는 일이었습니다. 원래는 안 되는 사람이 생기면 다시 메시지를 돌리고, 완곡하게 양보를 부탁하고, 눈치를 봐야 했습니다. 42는 판단의 근거와 함께 그 에너지까지 덜어줍니다. 완벽한 시간이 없어도 멈추지 않고, 최종 결정은 사람이 합니다.

🎬(서브클립 3분기) 확정(“괜찮은 시간”) → 성립 0(“다 되는 시간 없음 → 후보 시간 보기”) → 재조율(참석자 제안 → “가장 적게 흔드는 순서” → 자동 공지)

필수/선택 → 필참 전원 최선괜찮은 → “괜찮은 시간”성립 0 → 정직한 대안양방향 조율재메시지·완곡한 부탁 에너지 ↓

추천은 제안이고, 결정은 사람이 합니다.

09 · Iteration

완성한 뒤 평가하지 않고, 만드는 과정에서 계속 고쳤습니다

서비스를 처음 접한 3명이 동작하는 프로토타입을 사용하는 모습을 관찰했습니다. 발견한 문제는 심각도와 반복 여부, 수정 범위를 기준으로 비교해 핵심 흐름에 영향을 주는 항목부터 개선했습니다.

발견한 문제근거결정
가능한 시간이 없을 때 다음 행동이 없음기능 공백개선
‘참석 어려워요’가 버튼으로 보이지 않음행동 요소 인식 실패개선
회의 확정 상태가 명확하지 않음2/3 반복개선
필수·선택 참석자의 의미가 불분명함구조 검토 필요보류

사용자 피드백으로 바뀐 세 가지

  1. 01

    결과가 없어도 다음 선택지를 제공

    빈 화면 대신 가장 가까운 후보와 조정 가능한 조건을 제안했습니다.

  2. 02

    버튼의 행동 단서를 강화

    테두리와 배경을 추가해 ‘참석 어려워요’를 누를 수 있는 버튼으로 인식하도록 수정했습니다.

  3. 03

    확정 상태를 명확하게 표시

    확정 문구와 상태 표현을 강화하고 참석자의 확인 여부를 추가했습니다.

빠르게 구현하고 검증하며, 사용자 반응을 다음 버전에 바로 반영했습니다.

10 · Reflection

돌아보며

짧은 기간이 남긴, 정직한 배움들입니다.

빠르게 만든 대가

처음부터 바이브코딩으로 구현해 동작하는 프로토타입으로 빠르게 검증했지만, 디자인 툴이면 몇 번의 조작으로 끝날 미세조정과 세부 QA를 언어로 지시하느라 오히려 시간이 더 드는 순간도 있었습니다.

스코프를 좁히는 일의 어려움

'1시간짜리 회의를 잡는다'는 좁은 시나리오로 시작했지만, 더 쉬운 경험을 좇을수록 결국 워크 앱·캘린더·메신저가 맞물리는 영역까지 번졌습니다. 무엇을 다루지 않을지 정하는 게 가장 어려웠습니다.

구현하지 못한 범위

회의실 예약·전체 참석 확인(RSVP)·응답 마감·장소 공지는 사용자 테스트에서 필요를 확인했지만, 본질이 아니라 이번 범위 밖으로 두고 확정을 다음 단계로 넘어가는 지점으로만 다듬었습니다.

고려했지만 만들지 않은 것들 — 캘린더 자동 추천과 장소 예약

좋은 경험은 맥락에서 나온다

시간을 잡는 경험은, 뒷단에 사용자 데이터를 많이 가지고 있을수록 더 쉽고 편리해집니다. 좋은 스케줄링 경험은 결국 얼마나 많은 맥락 — 캘린더·일정·관계 — 을 아느냐에 달려 있었습니다.

데모 열어보기 설계 결정 로그 경쟁 리서치 디자인 시스템
